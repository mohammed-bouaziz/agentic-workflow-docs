Domain extractor
Step 0 — runs first, unblocked. Fast, cheap, deterministic.
Payload (varies per call)Runtime
{ "task": { "action": "extract_domain", "email": "james@uk.acmecorp.com", "company_name_raw": "Acme Corp Ltd" }, "output_schema": { "domain": "string", "confidence": "high|medium|low", "normalisation_note": "string|null" } }
System prompt (stable)Fixed
You extract the canonical domain from a raw email address. Rules: - Strip subdomains (uk., mail.) - Prefer the email domain over company name when they conflict - Return JSON only, no prose - If uncertain, set confidence to "low" and explain in normalisation_note
Design decisions
ScopeTwo fields only. No ICP, no CRM record, no enrichment data — this agent does one string operation. Anything else is noise that could cause it to "help" by inferring context.
FormatThe normalisation rules live in the system prompt, not the payload. They are stable across all calls. The payload carries only what changes: the raw email and name.
Output schemaConfidence and normalisation_note are explicit fields. Downstream agents that use this domain as a lookup key need to know whether to trust it or flag the lead for review.
Enrichment fetcher
Parallel track A — runs immediately after domain extractor. Calls Clearbit / Apollo.
Payload (varies per call)Runtime
{ "task": { "action": "fetch_firmographics", "domain": "acmecorp.com", "company_name": "Acme Corp" }, "scope": { "fields_required": [ "industry", "employee_count", "annual_revenue_usd", "hq_country", "tech_stack" ] }, "output_schema": { "industry": "string", "employee_count": "integer", "annual_revenue_usd": "integer|null", "hq_country": "string", "tech_stack": "string[]", "data_confidence": "high|medium|low" } }
System prompt (stable)Fixed
You fetch and normalise company firmographic data from external enrichment APIs. Return only the fields listed in fields_required. Do not infer or hallucinate missing values — use null and set data_confidence to "low" if a field is unavailable. Return JSON only.
Design decisions
Scopefields_required is explicit in the payload. Without it, the agent returns a full Clearbit record (40+ fields) and every downstream agent has to filter. Declaring the required fields here costs nothing and eliminates that cost everywhere else.
NecessityNo ICP definition here. The enrichment fetcher doesn't score — it fetches. Injecting ICP criteria risks biasing what it retrieves or normalises. Scoring logic belongs in the scoring agent's envelope only.
Couplingtech_stack is fetched here alongside firmographics because it uses the same API call. Fetching it separately in a second agent would mean a second API call to the same endpoint. Combine where the source is the same.
Research fetcher
Parallel track B — runs simultaneously with enrichment fetcher. Searches news + LinkedIn.
Payload (varies per call)Runtime
{ "task": { "action": "build_research_brief", "domain": "acmecorp.com", "company_name": "Acme Corp", "contact_email": "james@acmecorp.com" }, "scope": { "signals_to_find": [ "recent_funding", "product_launches", "executive_changes", "hiring_intent" ], "recency_window_days": 90 }, "output_schema": { "signals_found": [{ "type": "string", "summary": "string (max 2 sentences)", "source_url": "string", "date": "ISO date" }], "contact_role": "string|null", "contact_tenure_months": "integer|null" } }
System prompt (stable)Fixed
You research companies and contacts using web search and LinkedIn. Surface only signals relevant to a B2B sales context. Do not editorialize or score. Return structured signal objects, each with a max 2-sentence summary. Ignore signals older than recency_window_days. Return JSON only.
Design decisions
ResolutionThe output is resolved signals, not raw articles. The outreach drafter receives a structured array of signal objects, not a pile of URLs. Summarisation happens here so the drafter reasons about content, not format.
Scopesignals_to_find and recency_window_days are explicit constraints in the payload. Without them, the agent surfaces everything it finds and the outreach drafter has to filter. Scope constraints are cheaper to enforce at the source.
NecessityNo ICP definition, no scoring criteria, no email templates. This agent gathers — it does not evaluate. Injecting evaluation criteria here biases what it searches for.
Scoring agent
Runs after track A completes. Does not wait for track B — score is independent of research.
Payload (varies per call)Runtime
{ "task": { "action": "score_lead" }, "scope": { "firmographics": { "industry": "SaaS", "employee_count": 320, "hq_country": "US", "annual_revenue_usd": 18000000, "tech_stack": ["Salesforce","AWS"] }, "icp": { "target_industries": ["SaaS","Fintech"], "min_employees": 50, "max_employees": 1000, "target_geographies": ["US","UK"], "stack_fit_signals": ["Salesforce","HubSpot"] }, "existing_crm_record": null }, "output_schema": { "score": "integer 0-100", "tier": "hot|warm|cold", "reasons": "string[] (max 4)", "disqualified": "boolean", "disqualification_reason": "string|null" } }
System prompt (stable)Fixed
You score inbound leads against an ICP definition. Score 0-100 based on: industry fit, company size, geography, and tech stack alignment. If existing_crm_record is not null, set disqualified: true and return immediately. Return up to 4 short reasons explaining the score. Return JSON only, no prose.
Design decisions
FormatFirmographics arrive as a pre-filtered object with exactly the fields the scorer needs — not the full Clearbit payload. The enrichment fetcher already scoped this. The scorer reasons over named signals, not a document.
Timingexisting_crm_record is passed explicitly, even when null. The system prompt handles the null case — if a known customer, skip immediately. This is the short-circuit gate that prevents all wasted downstream work.
NecessityNo research brief, no news signals, no contact LinkedIn data. Scoring is a pure function of structured firmographic signals vs ICP criteria. Unstructured text degrades score consistency across calls.
Outreach drafter
Runs after score ≥ threshold AND research brief is ready. The only agent that needs both tracks.
Payload (varies per call)Runtime
{ "task": { "action": "draft_outreach_email", "to_email": "james@acmecorp.com", "lead_tier": "hot" }, "scope": { "company_snapshot": { "name": "Acme Corp", "industry": "SaaS", "employee_count": 320, "hq_country": "US" }, "signals": [{ "type": "recent_funding", "summary": "Raised $12M Series A in March" }], "contact": { "role": "VP of Engineering", "tenure_months": 8 }, "sender": { "name": "Sarah Chen", "title": "Account Executive", "cta": "book_demo" } }, "constraints": { "template_id": "growth_signal_opener", "max_words": 120, "personalisation_required": true }, "output_schema": { "subject": "string", "body": "string (plain text)", "personalisation_hook": "string" } }
System prompt (stable)Fixed
You write cold outreach emails for a B2B SaaS sales team. Tone: direct, consultative, no hyperbole. Never mention the score or internal signals explicitly. Always open with a specific, genuine reason for reaching out drawn from the signals provided. CTA mapping: book_demo → propose a 20-min call Use template_id as a structural guide, not a script. Return JSON only.
Design decisions
Resolutionsignals arrives as a pre-summarised array — not raw articles, not LinkedIn HTML. Each signal has a 2-sentence summary already produced by the research fetcher. The drafter reasons about what to say, not what the signal means.
Payload vs systemTone, the rule about not exposing scores, and CTA logic are stable across all calls — they live in the system prompt. What varies per call: company snapshot, signals, sender identity, template selection. These go in the payload.
Scopecompany_snapshot is 4 fields, not the full firmographic object. The drafter needs industry for tone calibration and size for context — not revenue figures or raw tech stack arrays it will never mention.
CRM writer
Runs last. A write agent — it stores final outputs, reasons about nothing.
Payload (varies per call)Runtime
{ "task": { "action": "write_crm_record" }, "scope": { "lead_email": "james@acmecorp.com", "domain": "acmecorp.com", "score": 82, "tier": "hot", "score_reasons": [ "Industry match: SaaS", "Stack fit: Salesforce", "Size within ICP range" ], "draft_email": { "subject": "Congrats on the Series A", "body": "..." }, "firmographics": { "industry": "SaaS", "employee_count": 320, "hq_country": "US" } }, "constraints": { "sequence_id": "outbound_hot_2024", "enqueue_email": true, "require_human_review": false }, "output_schema": { "crm_record_id": "string", "email_queued": "boolean", "queue_position": "integer" } }
System prompt (stable)Fixed
You write structured records to the CRM and enqueue emails in the outreach sequence. You do not evaluate, summarise, or rewrite any inputs. Map fields directly to CRM schema. If enqueue_email is true, add draft_email to sequence_id. If require_human_review is true, flag the record instead. Return JSON only.
Design decisions
NecessityNo research brief, no raw signals, no ICP definition, no sender persona. The CRM writer stores final outputs. It maps fields to schema. Every field in scope is a field it will write — nothing more.
Couplingfirmographics is included as a minimal 3-field snapshot because the CRM record needs it for lead enrichment fields. This is the same scoped subset passed to the outreach drafter — not the full Clearbit payload.
Payload vs systemenqueue_email and require_human_review are in the payload because they can vary per run configuration. The rules for what those flags mean are in the system prompt — stable logic, variable inputs.
